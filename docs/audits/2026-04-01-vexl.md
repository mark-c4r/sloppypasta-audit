## Audit Scope

**Surfaces assessed**: vexl.it landing page (including JS libraries and page source); vexl.it/faq; vexl.it/blog (7 posts including political censorship article); GitHub org (12 repos, TypeScript monorepo source, 28 releases, commit history); Play Store listing/reviews; published independent security audit; X/Twitter @vexl; App Store listing.
**Surfaces via proxy**: iOS app behavior from App Store listing and user reviews — TestFlight-only, not installed. App UI/UX from review text and FAQ flow descriptions. Firebase Cloud Messaging dependency from open-source code rather than runtime on de-Googled device.
**Surfaces not assessed**: Live app trading flow (mobile-only, not installed); leaderboard page (404); Foundation page (404); Financial Tyranny Index (archived); actual app notification behavior; behavior on GrapheneOS without Play Services; Vexl Next migration state.
**Limitations**: External audit of mobile-only product — UI quality, empty states, error handling cannot be directly observed. All Cat 3 scores from user reviews and FAQ. iOS TestFlight required invitation not obtained.

---

## Overall: 67% -- Grade C

Vexl is a mission-driven P2P Bitcoin trading app backed by SatoshiLabs (Trezor) and funded by grants from HRF and OpenSats. Its economic alignment is near-perfect: zero fees, zero ads, zero data monetization, pure donation/grant funding. The writing is clean with minimal AI markers, and the product's honesty about its business model is a standout. However, sovereignty scores are dragged down by structural centralization beneath the privacy-first surface -- phone number identity requirement (20% floor), single-server messaging dependency, no relay/node independence, and custodial key management with no export. The product is a platform dressed in protocol values. Design and product craft scores reflect the iOS TestFlight-only distribution, a 404'd leaderboard page, and generic release notes. The strongest signal: Vexl's economic model proves you can build useful Bitcoin software without extracting attention or data from users.

## Priority Improvements

1. **Identity Sovereignty (6.3): 20% -> 50%+**
   Allow keypair-based identity as an alternative to phone number. Even a secondary NIP-07 or pubkey-based login alongside phone would dramatically improve sovereignty scores. The phone requirement is the single largest score anchor in the entire audit.

2. **Relay/Node Independence (6.12): 25% -> 50%+**
   Publish a protocol specification for the marketplace and messaging layer. Allow third-party relays or at minimum let users configure custom server endpoints. The E2E encryption is excellent but worthless for independence if all traffic must route through Vexl's servers.

3. **Distribution Independence (6.9): 50% -> 75%**
   Publish to F-Droid and ship the iOS app through the App Store (not TestFlight). The TestFlight limitation is both a product craft problem (4.1) and a sovereignty problem. F-Droid with reproducible builds would align with the open-source commitment.

4. **Product Coercion (5.9): 40% -> 65%**
   Offer a web-based interface for browsing offers and managing trades, even if the mobile app remains primary. A PWA would satisfy both web access and reduce app store dependency simultaneously.

5. **Key Management Sovereignty (6.11): 25% -> 50%+**
   Allow users to export their identity keypair or connect external signing (NIP-46 pattern). The current model where Vexl generates and holds identity internally contradicts the sovereignty narrative.
