# Sloppypasta Audit: Sparrow Wallet

**Date**: 2026-04-01
**URL**: sparrowwallet.com
**Type**: Desktop app (Java-based Bitcoin wallet)
**Audit Mode**: External (V1, single-agent)
**Conditions**: [if:mobile], [if:identity], [if:payments], [if:content]
**Applicable Checks**: 45 of 46 (6.12 Relay/Node Independence N/A -- not a messaging/social product)

---

## Overall: 84% -- Grade B

Sparrow Wallet is the highest-scoring product audited in this round, narrowly missing an A grade. It is a textbook example of sovereignty-respecting software: fully non-custodial, keypair-based identity (100% on 6.3), open source under Apache 2.0, zero telemetry, zero tracking, and a donation-funded business model with no ads, no data monetization, and no attention harvesting. The writing across documentation and marketing is clean, human-authored, and free of LLM markers -- the developer Craig Raw writes with domain expertise and genuine voice. Economic alignment is near-perfect (95%), and the product's conduct toward users is exemplary: no access gates, no urgency tactics, no dark patterns, no nag campaigns.

What keeps Sparrow from an A grade is the Design & Interface category (71%), where a technically dense desktop UI -- while information-rich and appropriate for its power-user audience -- sacrifices some physical obviousness and accessibility basics. The UTXO-centric interface is intentionally complex, but new users face a steeper learning curve than necessary. The absence of an RSS feed for its published documentation (5.6: 30%) also drags the Product Conduct geometric mean down. These are minor weaknesses in a product that demonstrates what Bitcoin software should look like: a tool that serves the user, never surveils them, and can exist independently of any company.

## Priority Improvements

1. **RSS Feed (5.6): 30% -> 75%**
   Add an RSS/Atom feed for documentation updates and release notes at sparrowwallet.com/feed. For a project that champions open protocols and user sovereignty, the absence of RSS for published content is an ironic gap. This is the single lowest individual score in the entire audit and the easiest win available.

2. **Accessibility Basics (3.6): 55% -> 70%**
   Publish an accessibility statement. Audit the desktop app's contrast ratios in information-dense views (transaction hex, UTXO details). Ensure keyboard navigation works through all critical flows. JavaFX provides a11y primitives -- documenting and testing them would demonstrate care.

3. **Self-Evident UI (3.4): 65% -> 75%**
   Improve empty states for new wallets -- guide first-time users toward creating their first wallet with contextual help rather than requiring the external quick start guide. Progressive disclosure of advanced features (transaction editor, coin control) would reduce the learning curve without removing power.

4. **Physical Obviousness (3.2): 70% -> 80%**
   Add contextual tooltips or inline help for Bitcoin-specific concepts in the UI (UTXO, PSBT, coin selection algorithms). The documentation is excellent but the app itself could surface more guidance at the point of action.

5. **Reproducible Builds (6.9): 85% -> 95%**
   Implement and document reproducible builds. This would allow the community to verify that released binaries match the published source code -- the final link in the trust chain from source to installation.
